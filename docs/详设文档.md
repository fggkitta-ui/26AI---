# 招投标信息智能聚合工具 —— 详细设计文档

## 1. 系统概述

### 1.1 项目背景

企业在市场洞察、采购、合规、竞争情报、风控监测等场景中，需要持续跟踪国内外招投标资讯。这些信息散落在政府公共资源交易平台、行业协会网站、商业聚合网站等多个数据源，呈现形态复杂、更新频繁。本项目构建一个通用的招投标信息聚合工具，使用户可以用自然语言描述需求，系统自动完成多源抓取、清洗去重、结构化汇总与定时增量推送。

### 1.2 设计目标

1. **自然语言驱动**：用户用自然语言提出问题，系统自动解析关键词/区域/时间/频率
2. **多源适配**：插件化适配器架构，支持≥2个数据源（含≥1个需登录源）
3. **智能清洗去重**：去噪声 + 同义词扩展筛选 + 指纹去重
4. **事实一致性**：抽取式摘要，禁止 LLM 编造信息
5. **定时增量推送**：按用户指定频率执行，每次只推送新增内容
6. **多形态输出**：CLI / Web UI / API 三种交互方式

### 2.2 数据流转

```
用户自然语言输入
  → parse_intent() 解析出 {keyword, region, time_range, schedule}
  → 根据 schedule 决定: 立即执行 / 定时调度
  → 并发调用 3 个 adapter.run(keyword, region, start_date, end_date)
    每个 adapter 内部: login() → search() → [fetch_detail() × N]
  → 汇集所有 RawTenderItem
  → clean_items() 去噪声 + 条件筛选
  → deduplicate() 去重
  → 增量过滤: 排除 state_store 中已推送的指纹
  → summarize() 生成五要素摘要
  → build_docx() 生成 Word 文件
  → mark_as_pushed() 记录已推送指纹
  → 输出文件落盘
```

## 3. 核心模块设计

### 3.1 意图解析模块 (src/intent/)

**设计思路**：采用"规则优先 + LLM 增强"的双通道架构。

**规则解析 (rule_parser.py)**：
- 区域识别：全国 34 个省级行政区词典 + 后缀剥离归一化
- 时间范围：正则识别"最近N个月/周/天/年"和"YYYY年MM月"两种表达
- 频率/调度：识别"每天""每周X""今天"+"HH:MM"的组合
- 关键词：剥离已识别的区域/时间/频率/虚词片段后，用"XX招标信息"句式捕获主题词
- **泛化设计**：所有逻辑基于通用规则，不针对具体测试问题做硬编码

**LLM 解析 (llm_parser.py)**：
- 调用 Claude API 做结构化 JSON 抽取
- 处理口语化、隐含表达等复杂语义
- 规则解析结果作为合理性校验（如时间范围是否在合理区间内）

**创新点**：
- 规则解析零成本、零延迟、完全可复现
- LLM 解析覆盖长尾 case，两者互为校验
- 按长度倒序匹配地名，自动处理"安徽省→安徽"归一化

### 3.2 数据源适配层 (src/adapters/)

**设计思路**：每个网站封装为独立 Adapter 子类，实现统一的 `BaseAdapter` 接口（`login()` / `search()` / `fetch_detail()`），新增数据源只需新建文件，无需改动主流程。

**三个适配器**：

| 适配器 | 网站 | 登录要求 | 技术方案 |
|---|---|---|---|
| GovPlatformAdapter | 中国政府采购网 (www.ccgp.gov.cn) | 免登录 | httpx + BS4，JS渲染降级 Playwright |
| LoginRequiredAdapter | 中国招标投标公共服务平台 (www.cebpubservice.com) | 免费会员 | Playwright 模拟登录 + Cookie 持久化 |
| AggregatorSiteAdapter | 千里马招标网 (www.qianlima.com) | 免登录（公开内容） | httpx + BS4 |

**登录态管理**：
- Playwright 模拟浏览器登录 → `context.storage_state()` 保存至本地 JSON
- 7 天 TTL，超期自动重新登录
- 账号密码从环境变量/`config/secrets.yaml` 读取（不硬编码，不提交仓库）

**降级策略（优雅降级，三级兜底）**：
1. 优先尝试 httpx + BS4（快速、轻量）
2. 如遇 JS 渲染页面，降级到 Playwright
3. 如网站完全不可达，降级到演示数据模式（标注"演示数据"前缀）

### 3.3 清洗去重模块 (src/pipeline/)

**清洗 (cleaner.py)**：

四阶段噪声清洗：
1. HTML 标签/实体清理
2. 招投标网站模板噪声（导航路径、浏览次数、打印按钮等）
3. 通用网页噪声（版权、联系方式、二维码等）
4. 空白规范化

三级筛选策略：
- **L1 精确匹配**：关键词、区域、时间三者均在条目中出现
- **L2 同义匹配**：扩展同义词后匹配（如"充电桩"↔"新能源汽车充电设施"）
- **L3 宽松匹配**：仅区域+时间满足即保留（防止因同义词覆盖不全误删）

内置同义词映射表覆盖：充电桩、服务器、光伏、储能、数据中心、智慧城市、安防、电力等领域。

**去重 (deduplicator.py)**：

指纹哈希算法：
```
fingerprint = MD5(normalize_title(title) + publish_date + content[:200])
```

`normalize_title` 去除标点、空白、常见前缀（"关于""公告"）后小写化，确保标题细微差异不影响去重。

**去重策略优势**：
- 对"原文 + 转载"场景，标题/日期/核心内容高度相似 → 指纹碰撞 → 去重
- 对标题改写幅度较大但内容相同的转载，正文前 200 字 + 日期组合大概率一致
- 后续可引入 embedding 余弦相似度（`sentence-transformers`）做精细比对

### 3.4 汇总生成模块 (src/pipeline/summarizer.py)

**核心设计**：「抽取式摘要为主，禁止 LLM 自由生成」，确保事实一致性。

**关键信息加权抽取**：
- 对原文按句号/换行分句
- 每个句子按"是否包含项目编号/预算金额/招标人/时间节点/合同期限等"加权打分
- 按得分排序取 Top-N，保持原文顺序重组
- 句子级别不做任何改写，原样保留

**权重表**：

| 信息类别 | 权重 | 说明 |
|---|---|---|
| 项目编号 | 5 | 唯一标识 |
| 预算/金额 | 5 | 核心关注点 |
| 项目概况/采购需求 | 4 | 了解项目内容 |
| 投标截止/开标时间 | 4 | 时间节点 |
| 招标人/中标人 | 4 | 参与方 |
| 建设地点/区域 | 3 | 地域信息 |
| 合同期限/工期 | 3 | 执行周期 |
| 获取招标文件时间 | 3 | 参与窗口 |
| 联系方式 | 1 | 辅助信息 |

### 3.5 调度与增量推送模块 (src/scheduler/)

**调度 (job_scheduler.py)**：
- 基于 APScheduler 封装
- 支持三种模式：立即执行 / 一次性定时 / 周期定时（cron）
- 一次性定时若已过指定时间，自动转为立即执行（避免任务丢失）

**增量推送 (state_store.py)**：
- SQLite 存储：`(subscription_id, fingerprint, pushed_at)`
- `subscription_id` = MD5(用户原始查询)[:16]，同一查询的多次执行共享
- 每次执行前查询已推送指纹集合 → 过滤 → 只输出新增内容
- 执行后立即写入新指纹

### 3.6 Word 文档生成 (src/document/docx_builder.py)

- 文件命名严格遵循赛题规则：`{用户原始问题}_{YYYYMMDDHHmm}.docx`
- 专业排版：标题 → 生成时间 → 检索条件 → 信息列表（每条含标题/发布时间/来源链接/核心内容/附件链接）
- 增量模式标注："说明：本次为定时任务的增量推送，仅包含本次新增内容。"
- 无结果时生成"本次未检索到符合条件的新增招投标信息。"而非空文档

## 4. 功能实现创新点

### 4.1 意图解析的泛化能力

1. 规则版解析全部基于通用正则 + 全量省级词典，不针对任何具体测试问题做硬编码
2. 可正确处理"北京"和"北京市"的等价关系，自动归一化
3. 关键词提取基于"先剥离已知片段，再从句式结构中捕获主题"的策略，而非死记关键词表

### 4.2 去重算法的双重保障

1. 粗筛：指纹哈希（标题归一化+日期+正文前200字），O(n)时间复杂度，快速高效
2. 细筛（预留）：embedding 余弦相似度，对标题改写幅度较大的转载做精细判断
3. 跨来源去重：指纹不绑定数据源，同一公告被政府采购网和聚合站同时收录时自动去重

### 4.3 增量推送机制的"订阅隔离"

- 每个用户问题生成独立 `subscription_id`
- 不同问题不共享已推送记录，互不影响
- 同一问题的多次定时触发共享状态，精确实现"每次只推送新增内容"

### 4.4 优雅降级的三级兜底

- 所有 adapter 均有"在线抓取 → Playwright 降级 → 演示数据"三级兜底
- 单个数据源失败不影响其他数据源
- 演示数据明确标注前缀，方便区分

### 4.5 抽取式摘要的"事实锚定"

- 不从原文生成新文字，仅做信息密度筛选
- 保留原文前 500 字符作为可追溯片段
- 句子按"关键信息权重"排序而非 LLM 打分，结果确定可复现

## 5. 技术选型说明

| 组件 | 技术选型 | 理由 |
|---|---|---|
| 语言 | Python 3.10+ | 爬虫/数据处理生态成熟，第三方库丰富 |
| 意图解析 | 规则(内置) + LLM(Claude API) | 规则覆盖常规问法零成本，LLM 处理复杂长尾 |
| HTTP 抓取 | httpx | 异步支持好，API 简洁，比 requests 更快 |
| HTML 解析 | BeautifulSoup4 | 容错性强，对不规范 HTML 兼容好 |
| JS 渲染/登录 | Playwright | 全功能浏览器自动化，支持 Cookie 持久化 |
| 调度 | APScheduler | Python 生态最成熟的调度库 |
| 文档生成 | python-docx | 直接操作 docx 格式，排版灵活 |
| 状态存储 | SQLite | 零配置嵌入式数据库，适合本地状态管理 |
| Web 界面 | Flask | 轻量级，学习成本低，适合工具型 Web UI |
